## 1. ภาพรวมระบบ

WMS เป็นระบบติดตามและจัดการคิวรถบรรทุกที่เข้ามาโหลดสินค้า (กระเบื้อง / Fitting / Accessories) ในโรงงาน โดยมีเป้าหมายหลัก:

- **แสดงสถานะคิวรถแบบเรียลไทม์** — รถคันไหนรอคิว กำลังโหลด อยู่ขั้นไหน ใช้เวลาไปเท่าไร
- **แจ้งเตือนเมื่อมีรถติดอยู่ในขั้นตอนใดนานเกินกำหนด** (5 กฎ)
- **พยากรณ์เวลาที่รถจะโหลดเสร็จ** ด้วยโมเดล ML (XGBoost)
- **สรุปรายงานเชิงสถิติย้อนหลัง** (throughput, เวลาเฉลี่ย, overtime ฯลฯ)

### สถาปัตยกรรมระดับสูง

```
┌─────────────┐   REST (โหลดครั้งแรก)    ┌──────────────────────┐
│  Frontend   │ ───────────────────────► │   Backend (Django)   │
│  React+Vite │                          │   + Daphne (ASGI)    │
│             │ ◄═══════════════════════ │                      │
└─────────────┘   WebSocket (อัปเดตสด)   └───────────┬──────────┘
                                                     │
                          ┌──────────────────────────┼───────────────────────┐
                          ▼                           ▼                       ▼
                ┌──────────────────┐        ┌──────────────────┐    ┌──────────────────┐
                │  OBM_DWMS (อ่าน) │        │   WMS (เขียน log)│    │  โมเดล ML        │
                │  vwTimeStamp...  │        │ LogNotification  │    │ .joblib (XGBoost)│
                │  (ข้อมูลรถ/คิว)  │        │ PredictionLog    │    │                  │
                └──────────────────┘        └──────────────────┘    └──────────────────┘
```

หลักการสำคัญ:

- **OBM_DWMS** = ฐานข้อมูลต้นทาง (อ่านอย่างเดียว) — ข้อมูลรถ/ไทม์สแตมป์/สินค้าทั้งหมดมาจาก view `vwTimeStampDashbaord`
- **WMS** = ฐานข้อมูลสำหรับ "เขียน log" ของระบบเอง — เก็บประวัติการแจ้งเตือน (`LogNotification`) และผลพยากรณ์ (`PredictionLog`)
- ระบบ **ไม่มีตาราง Django model ของตัวเอง** — ใช้ raw SQL query ทั้งหมด ([api/models.py] ว่างเปล่า)
- การไหลของข้อมูล: Frontend โหลดครั้งแรกผ่าน REST แล้วสมัครรับอัปเดตผ่าน WebSocket; Backend คำนวณ snapshot เป็นรอบ ๆ แล้ว push ออกไป

## 2. Timeline รถบรรทุกตั้งเเต่เข้า - ออกโรงงาน (Truck Lifecycle)

นี่คือ "แกนกลาง" ที่ทุกโมดูลอ้างอิง — รถทุกคันจะเดินผ่าน **TimeStamp 6 จุด** (อ่านจากคอลัมน์ใน `vwTimeStampDashbaord`):

| ID | TimeStamp (DB column) | ความหมาย | ขั้นในไทม์ไลน์ |
|---|------------------------|----------------------------|----------------|
| 1 | `OperatorCarConfirm` | รถมาถึง/ยื่นตั๋วที่คลัง | เข้าโรงงาน |
| 2 | `CarConfirm` |รถถูกเรียกให้เข้าโหลด | เรียกคิว |
| 3 | `FirstPostPallet` | เริ่มโหลดพาเลทแรก | เริ่มโหลด |
| 4 | `LastPostPallet` | โหลดพาเลทสุดท้ายเสร็จ | โหลดเสร็จ |
| 5 | `CheckerClose` | เช็คเกอร์ตรวจปิดงาน | ปิดงาน |
| 6 | `PostingTime` | Post เข้าระบบ (รถออก) | ออกโรงงาน |

> **กฎสำคัญของ logic ทั้งระบบ:** "ขั้นถัดไปเริ่ม = ขั้นก่อนหน้าจบ" และระยะเวลาที่ใช้ในแต่ละขั้น = `เวลาขั้นถัดไป − เวลาขั้นนี้` — ใช้ทั้งในไทม์ไลน์ของ Queue, การคำนวณ notification และการ log duration

**หมายเหตุเรื่อง `exitDate`:** field `exitDate` จะแสดง `PostingTime` **เฉพาะเมื่อ** `PackListStatus = 'OPERATORCOMPLETED'` เท่านั้น (ดู [api/services/truck_queues.py](backend/api/services/truck_queues.py) บรรทัด 69-72) — เพื่อให้ไทม์ไลน์ "ออกโรงงาน" ติ๊กก็ต่อเมื่องานจบจริง

## 3. สถานะต่าง ๆ ในระบบ (Status Logic)

ระบบมีสถานะหลายชั้น แต่ละชั้นคำนวณจากไทม์สแตมป์/ข้อมูลสินค้า ดังนี้

### 3.1 สถานะรถ (Truck Status)

มาจากคอลัมน์ `TruckStatus` ใน DB แปลงเป็นภาษาไทยตอน query:

| ค่าใน DB (`rawTruckStatus`) | แสดงผล (`status`) | ความหมาย |
|------------------------------|--------------------|----------|
| `Waiting` | **รอคิว** | อยู่ในคิว ยังไม่ assign ลาน (badge เหลือง) |
| `Loading` | **กำลังโหลด** | กำลังโหลดสินค้าอยู่ (badge ฟ้า) |

ฝั่ง frontend มีตัว normalize alias ([queue/constants/index.js](frontend/src/features/queue/constants/index.js)) ให้ค่า `loading`↔`กำลังโหลด`, `waiting`↔`รอคิว` เป็นค่าเดียวกัน เพื่อให้ filter/search ทำงานได้ไม่ว่าจะรับค่ามาแบบใด

### 3.2 ชนิดคิว (Queue Type)

คำนวณจาก `PickListType` + `PrepareForward` (logic ใน `QUEUE_TYPE_CASE`):

| เงื่อนไข | ชนิดคิว |
|----------|---------|
| `PickListType = 'SmartQ'` | **SmartQ** (จองคิวล่วงหน้าผ่านระบบ) |
| `PickListType = 'Walk-in'` และ `PrepareForward = 'N'` | **Walk in** (มาหน้างาน) |
| `PickListType = 'Walk-in'` และ `PrepareForward = 'Y'` | **ล่วงหน้า** (เตรียมของไว้ล่วงหน้า) |

### 3.3 สถานะการจัดสินค้าแต่ละหมวด (Component Status)

สำหรับสินค้า 3 หมวด — **กระเบื้อง (Tile)**, **Fitting**, **Accessories** — แต่ละหมวดมี 4 สถานะ คำนวณจากการมี/ไม่มีไทม์สแตมป์ `Start`/`End` ของหมวดนั้น (logic ใน [truck_queues.py](backend/api/services/truck_queues.py) บรรทัด 81-147):

| สถานะ | เงื่อนไข | ความหมาย | สีที่แสดง (frontend) |
|-------|----------|----------------------|----------------------|
| **ไม่มีรับ** | ยอดสินค้าหมวดนั้น = 0 | ไม่มีรับของในหมวดนั้น | เทา |
| **ยังไม่เริ่ม** | มียอด แต่ `Start IS NULL` และ `End IS NULL` | มีรับของ เเต่ยังไม่เริ่มจัด | แดง |
| **เริ่มจัด** | `Start IS NOT NULL` แต่ `End IS NULL` | เริ่มจัดเเล้ว เเต่ยังไม่เสร็จ | เหลือง |
| **เสร็จแล้ว** | `Start IS NOT NULL` และ `End IS NOT NULL` | จัดเสร็จแล้ว | เขียว |

- **Tile** ดูจาก `TileStart`/`TileEnd` (และต้องมียอด CPAC/PRESTIGE/NEUSTILE)
- **Fitting** ดูจาก `FittingStart`/`FittingEnd` (ยอด CPAC/PRESTIGE/NEUSTILE/DURA Fitting)
- **Accessories** ดูจาก `AccStart`/`AccEnd` (ยอด `ACCESSORIESSapAmount`)

### 3.5 สถานะช่องโหลด (Yard / Channel Status)
แต่ละช่องโหลด (post location) มีสถานะ:

| สถานะ | ความหมาย |
|-------|----------|
| **ว่าง (available/empty)** | ไม่มีรถกำลังโหลด |
| **กำลังโหลด (loading)** | มีรถกำลังโหลดอยู่ |

โดยพิจารณาความ active ของรถยกจาก `DEFAULT_FORKLIFT_ACTIVE_MINUTES` (= 5 นาทีล่าสุดมีกิจกรรม ⇒ ถือว่ายัง active)

### 3.6 สถานะพยากรณ์ (Prediction Status)

| สถานะ | ความหมาย |
|-------|----------|
| **in_progress** | รถยังโหลดอยู่ (ยังไม่มี `PostingTime`) — มีค่าพยากรณ์ แต่ยังไม่รู้ค่าจริง |
| **completed** | รถจบงานแล้ว (มี `PostingTime`) — คำนวณ error ได้แล้ว |

## 4. ระบบแจ้งเตือน (Notifications)

นี่คือส่วนที่ logic ซับซ้อนที่สุด แบ่งเป็น 2 ส่วน: **การคำนวณว่าจะเตือนอะไร** กับ **การบันทึก log ลง DB**

ไฟล์: [api/services/notification_service.py](backend/api/services/notification_service.py) — `compute_notifications()`

แต่ละกฎผูกกับ "ช่วงเวลาระหว่าง 2 ไทม์สแตมป์" — ถ้ารถ **เริ่มขั้นนั้นแล้ว แต่ยังไม่ขึ้นขั้นถัดไป** และ **เกิน threshold** ⇒ เตือน

| ID | เเจ้งเตือน | ความหมาย | เริ่มจับเวลาที่ | สิ้นสุดเมื่อ | Threshold |
|---|---------|---------|-----------------|--------------|-----------|
| 1 | **รอเรียก** | เเตกตั๋วเเล้ว เเต่ยังไม่ถูกเรียกเข้าลาน | `OperatorCarConfirm` | `CarConfirm` | 5 นาที |
| 2 | **รอโหลด** | เรียกเข้าลานแล้ว แต่ยังไม่เริ่มโหลด | `CarConfirm` | `FirstPostPallet` | 15 นาที |
| 3 | **โหลดนานเกิน** | ใช้เวลาโหลดเกิน 45 นาที | `FirstPostPallet` | `LastPostPallet` | 45 นาที |
| 4 | **รอปิดงาน** | โหลดเสร็จแล้ว แต่ยังไม่ปิดงาน | `LastPostPallet` | `CheckerClose` | 5 นาที |
| 5 | **รอ Post** | ปิดงานแล้ว แต่ยังไม่ post | `CheckerClose` | `PostingTime` | 5 นาที |
